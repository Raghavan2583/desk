DESK — DEpendency riSK · Engineering Deep Dive

How DESK actually works, and why it's built this way.

Built with AI assistance, and written the way I'd defend it myself — the architecture, the data model, the trade-offs, and the production incidents that actually happened, including the two times I built something and later removed it.

01

Project Story & Business Understanding

The elevator pitch

"DESK maps risk across the PyPI dependency ecosystem. It answers one question for any of the top 1,000 Python packages: if this package's maintainer disappeared tomorrow, or a critical CVE dropped in it today, how much of the ecosystem would be affected — and how would I know before it happened?"

Why this problem exists

Modern software is a thin layer on top of hundreds of transitive open-source dependencies, most maintained by unpaid volunteers with no SLA. Three concrete failure modes motivate DESK:

  1. Maintainer abandonment — a package quietly stops getting commits. Nobody notices until a CVE goes unpatched for months.
  2. Silent CVE exposure — a critical vulnerability lands three dependency layers deep and no dashboard surfaces it.
  3. Blast radius blindness — you don't know that urllib3 going down would break 200+ other packages, so you can't prioritize which risks actually matter.

No existing tool scores the whole ecosystem this way. Tools like Snyk/Dependabot are project-first and reactive — they scan your requirements.txt after the fact. DESK is ecosystem-first and proactive.

Who uses it, and the value

Engineering leads doing dependency risk triage, security teams doing OSS due diligence, platform teams deciding which packages to standardize on. The value is one explainable score (0–10, labeled LOW/MEDIUM/HIGH/CRITICAL) with a visible component breakdown — not a black box — plus a visual blast-radius graph so impact is intuitive.

In one sentence

"I built a data pipeline that ingests signals from four independent sources — PyPI, GitHub, a vulnerability database, and dependency graph data — for the top 1,000 PyPI packages, transforms them through dbt into a dimensional model, computes a weighted risk score with an explainable formula, and serves the result as a static, pre-computed dependency graph — no live backend, no per-request compute."

Communication between components

There is no network call between pipeline stages — each stage reads/writes the same DuckDB file (data/desk.duckdb) on the GitHub Actions runner's local disk, sequentially, in one job. That's only safe because everything runs in one job, nothing else writes concurrently, and the file is ephemeral and gitignored. What persists across runs is committed to git: three Parquet history files and the exported JSON. Git is the persistence layer here, not a database.

Advantages and limitations

Advantages: zero infra cost, zero DB ops, full historical audit trail via git log, zero cold start / zero backend attack surface on the frontend.

Limitations — own these first:

  • Not real-time — up to 24h stale by design (an hourly event-trigger for this was actually built, then deliberately removed — see Section 12)
  • Single sequential job — if one ingestion step hangs, everything downstream waits (bounded by a 60-minute timeout)
  • DuckDB is single-writer, single-machine — fine at 1,000 packages, would need redesign at 100,000+
  • Git-as-storage grows ~5MB/year in Parquet history — fine for years, a long-term watch item

Walk one record end to end: requests

  1. PyPI ingest — fetches info.name, version, requires_dist, project_urls. Finds github.com/psf/requests via Source. Downloads are rechecked every package, every run (D064) → pypistats gives monthly_downloads = 98,000,000. Row → raw_pypi_packages.
  2. GitHub ingest — batched GraphQL alias query against psf/requests: pushedAt, 90-day commit count, isArchived: false. Row → raw_github_maintainers.
  3. OSV ingest — phase 1 batch query returns known vuln IDs; phase 2 fetches full CVSS detail per unique ID (say, one historical HIGH, already patched). Row(s) → raw_osv_cves.
  4. deps_dev ingest (really: local requires_dist parser) — keeps only deps also in the top-1,000, flags ; extra == deps as optional. urllib3, certifi, charset-normalizer, idna become edges; each target's dependent_count increments.
  5. dbt staging — dedups to latest row; computes days_since_last_commit and activity_label = 'ACTIVE'.
  6. dbt intermediate — LEFT JOINs PyPI + GitHub on github_repo_url, sets has_github_link = TRUE; aggregates CVE counts and blast radius.
  7. dbt marts — one dim_packages row, four outbound fact_dependencies edges.
  8. Scoring — resolves maintainer_status = LIVE and downloads_status = LIVE. M_raw 0.0 (committed ≤30d) · C_raw small (one patched CVE) · D_raw 10.0 (blast radius >200 → component 2.0) · DL_raw small. Final LOW ≈ 1.8 — blast radius is high, but that's only 20% of the formula.
  9. Export — node in graph.json plus its own package/requests.json with full component breakdown, patched CVE history, 12-month trend, top-10 dependents.
  10. Frontend — search resolves via preloaded index.json; clicking the node lazy-loads package/requests.json. No API call, no server round trip beyond the static asset fetch.

The one piece of logic worth memorizing cold

Part of every package's risk score depends on maintainer health — is a real person still actively looking after it? That means calling GitHub's API once a day for every package. One day, that call failed for a batch of packages — a rate-limit issue, nothing to do with the packages themselves. When data is missing, the code still had to put some number into the score, and the original logic filled in the worst possible value, as if "no data" meant "assume abandoned." Here's what that one shortcut did in production:

A GitHub GraphQL batch failure caused missing maintainer data for python-multipart. The scoring code defaulted missing maintainer health to the worst-case score (10.0/10 — "assume abandoned"). That single defaulting decision spiked the package from 3.9 to 7.9 and flipped 42 packages to CRITICAL that weren't actually critical — pure data-gap noise dressed up as a real signal.

The fix wasn't "pick a better default" — the middle-ground option (default to a neutral value instead of worst-case) was tried and rejected too, because averaging in ignorance still quietly biases every score, just toward "looks safe" instead of "looks dangerous." The actual fix was to stop guessing entirely. Every score now carries an honest label telling you exactly how trustworthy that number is, alongside the number itself:

LIVE CARRIED_FORWARD NEVER_VERIFIED NO_GITHUB_LINK
  • LIVE — today's check actually succeeded. This number is fresh and real.
  • CARRIED_FORWARD — today's check failed, but a real value exists from a previous successful check. Show that real number, with its real date — never stamp it with today's date.
  • NEVER_VERIFIED — we've never once successfully gotten real data for this package. Say so directly, rather than showing a confident-looking score that's actually a guess.
  • NO_GITHUB_LINK — a different case entirely: no repository exists to check. Scored neutral by design, not because data is missing.

Implemented in risk_score.py::_resolve_maintainer / _resolve_downloads

This same pattern was independently re-applied to download counts (D059/D061) once the team recognized pypistats.org's rate limit meant not every package could get a live check per run — though the rotation workaround it originally required was later found unnecessary and removed entirely (D064, see Section 12).

Why this matters

There's a real difference between "missing data" and "bad data" — defaulting to honest uncertainty over confident guessing is the harder, more senior instinct, and it only came from actually getting burned by the alternative once.

Data Modelling Deep Dive

The layering

medallion-style layeringdbt/models/
desk_raw  (Bronze — append-only, one row per ingest run per entity)
    │
staging   (clean: dedup, type-cast, no business logic)
    │
intermediate  (join, aggregate — entity resolution)
    │
marts     (Silver/Gold — query-ready dimensional model)
    │
fact_risk_scores + fact_risk_score_history  (scoring output, NOT built by dbt)

Entities and relationships

  • dim_packages (dimension, PK package_name) — one row per package.
  • dim_maintainers (dimension, PK github_repo_url) — keyed by repo, not package, because the model correctly avoids assuming a 1:1 package↔repo relationship (monorepos exist).
  • fact_dependencies (edge fact, grain: package → dependency pair) — classic many-to-many fact table.
  • fact_risk_scores (current snapshot, truncate-and-replace) vs. fact_risk_score_history (append-only, grain: package × computed_at) — a Type-2-slowly-changing pattern implemented as current-snapshot + append-only-history, rather than literal valid_from/valid_to columns. Simpler for a daily batch use case where you only ever want "now" or "the full timeline."

Why this schema and not another

Why split dim_packages and dim_maintainers? A package can have has_github_link = FALSE. Splitting means "no maintainer data" is a natural LEFT JOIN NULL, not a table full of sentinel values.

Why FULL OUTER JOIN, not LEFT JOIN, in int_dependency_metrics?

sqldbt/models/intermediate/int_dependency_metrics.sql
FROM dep_edges e
FULL OUTER JOIN {{ ref('stg_deps_dependents') }} d
    ON e.package_name = d.package_name

Packages with zero outgoing dependencies but nonzero incoming dependents (leaf packages like certifi, six, idna) would silently disappear from a LEFT JOIN FROM dep_edges — the SQL comment states explicitly that a LEFT JOIN would "silently zero-out their depth component." A real bug anticipated and designed around, not discovered in production.

Data warehouse concepts present

  • Star-schema-like mart layer: facts (fact_dependencies, fact_risk_scores) + dimensions (dim_packages, dim_maintainers)
  • Slowly-changing history via append-only fact + current-snapshot fact
  • Raw layer is append-only and immutable — classic "raw zone never gets UPDATEd" even though the engine (DuckDB) isn't a data lake
  • Idempotent, full-rebuild transforms — no incremental merge logic, the right call at this volume
  • Deliberate shift from normalized (marts, for correctness) to denormalized (per-package JSON, for read performance) — the classic OLTP/OLAP/serving-layer tradeoff

Code-Level Understanding

Design patterns actually in use

  • Retry-with-backoff as a higher-order function (backoff.py) — wraps any callable with exponential backoff + jitter, distinguishing retryable (429/5xx/timeout) from non-retryable (other 4xx) failures. A retry policy, not a full circuit breaker — a reasonable scope call for a daily batch job.
  • State machine — scheduler_queue.status: pending → running → complete/error, giving every run a full audit trail.
  • Token pool with a dormant scale-out seam — TokenPool.rotate() exists but is never called in MVP. Build the seam now, wire it up later — visible forward design without premature complexity.
  • Provenance tagging — the *_status fields are an honest-null pattern: never let a consumer confuse "we checked and it's fine" with "we didn't check."
  • NaN sanitization at the JSON boundary — _nan() uses the NaN != NaN identity trick, paired with allow_nan=False so a stray NaN raises loudly at export time instead of shipping invalid JSON to the browser. Fixed twice — first pass covered raw rows, second pass discovered pandas .df() reverts None back to NaN on read. A good story about incomplete first fixes and verifying assumptions.

Error handling — three tiers, matched to blast radius

  1. Per-request retry (transient network / rate limits)
  2. Per-package isolation (one bad package doesn't kill the run; marked error and skipped)
  3. Per-source pipeline isolation (continue-on-error on the OSV step only)

Notably, not everything is soft-failed: dbt build/dbt test failures hard-fail the pipeline, because bad data reaching fact_risk_scores is worse than a delayed run. Soft-fail on ingestion, hard-fail on data-quality gates — a deliberate asymmetry.

Configuration & logging

Environment-variable driven throughout (DESK_DB_PATH, RISK_SCORE_VERSION, GITHUB_TOKENS...) — no hardcoded magic numbers. RISK_SCORE_VERSION is a first-class knob specifically because bumping the formula must invalidate trend comparisons. Structured leveled logging (logging, never print()) with greppable context, e.g. SCHEMA_CHANGE_DETECTED source=%s field=%s, plus progress logging every 100 items so a hung run is diagnosable mid-flight.

System Design Thinking

"How would this handle 10x growth (10,000 packages)?"

Honest answer: it breaks first at the GitHub Actions 60-minute timeout and GraphQL rate limits, not at DuckDB.

  • Ingestion — scheduler_queue.priority already exists (currently unused). At 10x, priority-tiered refresh becomes real: top packages daily, the rest weekly on rotation — extending the same pattern already proven for download counts.
  • GitHub API — TokenPool.rotate() already exists, dormant. Multiple PATs round-robin'd multiply the effective rate ceiling.
  • Orchestration — one 60-minute job becomes sharded parallel jobs (GHA matrix builds), merged before transform. This is the point a real orchestrator starts earning its cost.
  • DuckDB — still fine at 10,000 rows/day; becomes a real constraint near millions of packages, where a distributed warehouse is the right call.

"Batch vs. streaming — would you ever go streaming?"

No, and defending that "no" is the actual skill being tested. Commit cadence, CVE publication, download trend don't change on sub-daily timescales. An hourly event-driven trigger for this was actually built and shipped — then removed (D065) once it became clear nobody was relying on same-day freshness yet. The right amount of real-time here turned out to be zero, once tested against actual usage instead of assumed in advance. Full story in Section 12.

Partitioning, caching, monitoring, security, availability

  • Partitioning — raw tables conceptually partitioned by ingested_at
  • Caching — the static JSON export is the cache, served from Vercel's CDN edge
  • Monitoring — weekly proactive schema drift check + GitHub's default failure emails; real alerting (Slack/webhook) is a named next step
  • Security — post-DuckDB migration there are no database credentials at all — smaller attack surface by construction
  • Availability — frontend availability = Vercel's CDN uptime, fully decoupled from pipeline availability; a failed run means stale data, never downtime

The one alternative worth raising unprompted

"Why not just scan requirements.txt like Snyk/Dependabot?" — because that's reactive per-project scanning; DESK is proactive per-ecosystem scoring. They're complementary: Dependabot tells you about CVEs in packages you already use; DESK tells you which packages are becoming risky before you add them, by scoring maintainer health as a leading indicator.

Common Questions

How do you determine the top 1,000 packages?

From hugovk's top-pypi-packages dataset — a public, regularly-updated ranking of PyPI packages by download volume over the trailing 30 days. The pipeline re-fetches this list and re-seeds the package queue from it on every single run, so the roster of "top 1,000" isn't fixed at launch — it shifts over time exactly as real-world download patterns shift.

Is there a backend?

Not at request time — no. The site you're looking at is a static React app reading pre-computed JSON files from Vercel's CDN; there is no server answering your search queries. What does the work instead is a scheduled batch pipeline (GitHub Actions, once daily) that ingests from four APIs, transforms the data with dbt, computes risk scores, and exports the result as static files. That pipeline is a real backend in the "does compute, owns data" sense — it's just not one that serves live requests. A request-serving backend would only become necessary if DESK needed live per-user state (accounts, saved watchlists), on-demand scoring of packages outside the top 1,000, or push notifications — none of which are in scope right now.

Walk me through the architecture.

Four independent ingestion scripts (PyPI, GitHub, OSV.dev, and a local dependency-graph derivation) write into DuckDB. dbt transforms that through staging → intermediate → mart layers. A Python scoring script reads the marts and computes a weighted risk score. An export script turns the scored data into static JSON. The whole thing runs once a day as one GitHub Actions job — see the diagram in Section 2.

Why DuckDB instead of a traditional database?

Zero infrastructure — it runs in-process inside the CI job, with no server, account, or billing to manage. It replaced an original BigQuery design specifically to remove GCP credential-management overhead (D057). The trade-off is no concurrent multi-writer access and no persistence across runs without an explicit export step — both non-issues here, since the pipeline is single-writer and strictly sequential.

How does the pipeline handle failures?

Three tiers: per-request retry with backoff, per-package isolation (one bad record doesn't take down the batch — D020), and per-source isolation for the flakiest API. dbt test failures are the one place that hard-fails the whole run on purpose — bad data reaching the scoring layer is worse than a delayed run.

How do you maintain data quality?

Schema validation at the ingestion boundary, a weekly proactive schema-drift check against all four upstream APIs, dbt schema tests on every mart table, and the LIVE / CARRIED_FORWARD / NEVER_VERIFIED provenance pattern (Section 3) so a missing value is never silently treated as a real one.

Why static JSON instead of a live API?

The data changes once a day. A live API answering a constant 24-hour question is overhead with no upside — every query shape the frontend needs is already known ahead of time and pre-computed at export.

Why not make this real-time end to end?

The underlying signals — commit cadence, CVE publication, download trend — don't change on a sub-daily timescale. An hourly event-trigger for high-traffic packages was actually built and run in production for two months, then removed (Section 12) once it was clear nobody needed same-day freshness yet. "No" was tested against real usage, not just assumed.

What are the current limitations?

Up to 24 hours of staleness by design. Single-sequential execution — no horizontal scale-out yet. Git-as-persistence for history files, which won't scale indefinitely. DuckDB is single-machine. Blast radius only counts direct (depth-1) dependents, not the true transitive chain. Each of these has a designed-but-dormant seam for the point it'd actually need fixing (Section 8, Section 11) — not a gap that was missed, a line that hasn't been crossed yet.

Tell me about a bug you found and fixed.

The false-CRITICAL missing-data incident (Section 7) is the sharpest one: a GitHub ingestion failure silently spiked 42 packages to CRITICAL because missing data defaulted to a worst-case score. Caught by reviewing pipeline output and cross-checking it against expectations, not by an automated alert — that's a real gap, not a solved one. Two others worth knowing: a GraphQL "resource limits exceeded" response logged as a warning instead of raised, silently losing maintainer data for 762 of 900 packages in one run; and a git-push race condition between two workflows writing to the same branch, fixed with a shared concurrency group.

Tell me about a time you removed something you built.

The hourly event-trigger (Section 12) — it worked, it was real engineering, and it still got cut. No live usage depended on it, it had already caused a real concurrency bug, and the daily pipeline already covered everything it watched for. Keeping it would have mainly demonstrated "this can build event-driven triggers" rather than solved a live problem — the weakest possible reason to keep a piece of infrastructure around.

How would this scale to 10,000 packages?

It breaks first at the GitHub Actions 60-minute timeout and GraphQL rate limits, not at DuckDB. The fixes are already designed as dormant seams rather than built: scheduler_queue.priority for tiered refresh (top packages daily, the rest on rotation), TokenPool.rotate() for multi-token GitHub throughput, and sharded parallel ingestion jobs merged before transform.

11

Retrospective

What I'd keep

  • DuckDB over BigQuery — the infra/ops trade-off was correct for the actual concurrency needs here, which is none
  • Labeling every score with how trustworthy it is — LIVE, CARRIED_FORWARD, or NEVER_VERIFIED (Section 3) — instead of quietly guessing when data is missing. This came directly from a real incident where a missing-data guess caused a false alarm, and fixing it properly (rather than picking a slightly-less-wrong guess) is the single decision I'd point to first as production-grade thinking.
  • Deriving dependencies from requires_dist instead of depending on deps.dev's broken endpoint — a pragmatic adaptation when an external dependency didn't work as advertised
  • Retiring the download-count rotation and the hourly event-trigger (Section 12) — both were correct calls once tested against reality, not mistakes to walk back from

Trade-offs still standing, and why

  • Daily batch freshness over real-time — correct for signals that don't move sub-daily; would only revisit if the scope expanded to faster-moving data
  • Single-sequential-job pipeline — simplest thing that works at 1,000 packages; no parallelism yet because nothing has demanded it
  • Blast radius only counts direct (depth-1) dependents within the top 1,000 — true transitive blast radius needs a working deps.dev integration or a recursive resolution pass, neither built yet
  • No real alerting beyond GitHub's default failure emails — a known gap, not an oversight

What I'd build next, in priority order

  1. Wire up scheduler_queue.priority and TokenPool.rotate() — both already designed, neither load-bearing yet
  2. Real alerting (Slack/webhook on failure) instead of GitHub's email default
  3. Transitive blast radius (depth 2+) — the current model explicitly doesn't attempt this
  4. A "what changed since yesterday" diff view — the Parquet history already contains everything needed; it's a query away, not a redesign

"I designed the data model, the ingestion resilience strategy, and the scoring formula's edge-case handling — including redesigning it after a real incident taught me that defaulting missing data to a 'safe' or 'neutral' value is itself a bias, not a fix. I made and documented the storage migration from BigQuery to DuckDB when the ops cost stopped being worth it, and I removed two pieces of my own working infrastructure once the evidence said they weren't earning their keep. Every decision here has a reason behind it — including the ones to undo an earlier decision."

Recent Iteration — Reversing My Own Decisions When the Evidence Said So

The two stories below both happened in the same week. Neither is a "we shipped a feature" story — they're "we built something real, then had the discipline to correct it once the evidence came in" stories. That's a different skill than building, and it's the one most portfolios don't show.

The download-rotation saga — three attempts to the same fix

D061 — built a workaround for a limit that wasn't real. pypistats.org's download-count endpoint has a 30-requests-per-minute limit. With 1,000 packages to check and no obvious way around it, the fix was a rotation: check ~40 least-recently-checked packages per run, full rotation every 3-4 weeks, carry the last real value forward the rest of the time. It worked. It shipped.

D063 — found a bug hiding inside the fix. Reviewing the code a day later, the rotation's "is this package due for a recheck" logic ran after an unrelated "skip if the PyPI version hasn't changed" check. A package due for its rotation slot could get skipped anyway if its version happened to be unchanged — silently re-stamping a stale download count as fresh. Two independent pieces of "skip this work" logic had quietly combined into "skipped more often than either one intended." Fixed by reordering the checks.

D064 — realized the original constraint was mis-read, and undid the whole workaround. Re-examining the real numbers: 30 requests/minute is 1,800 requests an hour. Pacing 1,000 requests 3 seconds apart takes about 50 minutes — comfortably inside the limit. The actual constraint was never "you can't check 1,000 packages a day," it was "don't burst them all in 4 seconds," which the original code had done by accident. The entire rotation mechanism — the thing built in D061 and just carefully debugged in D063 — got deleted. Every package's download count is now rechecked every single day.

Why this is worth telling

The instinct to defend a fix once you've built and debugged it is real — by D063 there was already sunk effort in making the rotation correct. The harder, better move was going back to first principles on the constraint itself, discovering the workaround was solving the wrong problem, and deleting it rather than polishing it further.

Retiring the hourly event-trigger (D065)

pypi_event_trigger.yml ran every hour, checking PyPI's RSS feed for new releases of ~50 high-traffic packages (requests, numpy, django, and similar). If one had a new version, it triggered the full pipeline immediately — so a change to a widely-used package showed up within the hour instead of waiting for the next 02:07 UTC daily run. It had already caused a real production bug (D034 — two workflows writing to the same git branch without coordination, fixed with a shared concurrency group) and had been running in production for two months.

Reviewing it fresh: the daily run already covers all 1,000 packages, including those same 50, once every 24 hours regardless. The hourly job's entire value was shrinking the staleness window from 24 hours down to about one — for a project where no one is refreshing the page hourly waiting for that difference. Nobody was relying on it. Keeping it mostly proved "this can build event-driven triggers," which is the weakest possible reason to keep a piece of infrastructure that had already caused one bug.

It was removed — the workflow file deleted, every doc that described it as current state corrected, the decision logged as D065, and the 511 leftover GitHub Actions run records for it cleaned out of the repo's history.

The actual lesson

Not every real capability is worth keeping. The question isn't "does this work" — the rotation worked, the trigger worked — it's "is anything actually depending on this, right now." When the honest answer was no twice in the same week, the right move both times was deletion, not defense.
